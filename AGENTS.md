# AGENTS.md

## 1. 项目定位

本项目是一个独立的 MES Agent 研究项目。

当前技术栈：

- 前端：Vue 3 + Vite + JavaScript
- 后端：Python + FastAPI
- 模型：后续接入 DeepSeek
- 数据来源：后续通过只读接口或只读数据库访问 MES 数据
- 部署：本地开发优先，后续可使用 Docker Compose

核心原则：

- 先完成最小闭环，再扩展能力
- 数据正确性优先于智能化
- 业务规则必须确定、可验证
- 大模型只负责理解、编排和表达，不负责替代业务规则
- 禁止为未来需求提前设计复杂框架

---

## 2. 全局执行规则

任何任务开始前，必须先完成以下判断：

1. 当前需求是否明确
2. 修改范围是否清晰
3. 是否会影响现有功能
4. 是否需要新增依赖
5. 是否存在更简单的实现方式

执行时必须遵守：

- 只修改当前任务涉及的文件
- 不擅自扩大需求范围
- 不重构与当前任务无关的代码
- 不创建未被实际使用的抽象
- 不引入当前阶段不需要的中间件
- 不为了“以后可能使用”创建空模块
- 不隐藏错误，不吞异常
- 不伪造验证结果
- 不执行 Git commit
- 每次任务完成后更新 `log/codex-task-log.md`

## 4. Python 后端分层约束

Python 后端必须保持清晰分层。

推荐基础结构：

- `backend/app/main.py`
- `backend/app/api/`
- `backend/app/application/`
- `backend/app/domain/`
- `backend/app/infrastructure/`
- `backend/app/schemas/`
- `backend/app/core/`
- `backend/tests/`

各层职责如下。

### 4.1 main.py

`main.py` 只负责：

- 创建 FastAPI 应用
- 注册路由
- 注册中间件
- 注册生命周期事件
- 加载基础配置

禁止在 `main.py` 中：

- 编写业务逻辑
- 编写 SQL
- 调用大模型
- 实现 Agent 流程
- 处理复杂数据转换
- 定义大量接口

`main.py` 应保持短小，只作为应用入口。

### 4.2 api 层

`api/` 只负责 HTTP 协议相关内容：

- 接收请求
- 参数校验
- 调用 application 层
- 返回响应
- 转换 HTTP 状态码

禁止：

- 在路由函数中写业务判断
- 在路由函数中直接访问数据库
- 在路由函数中直接调用 DeepSeek
- 在路由函数中写复杂循环
- 在路由函数中拼装大段提示词
- 在路由函数中处理事务

路由函数应尽量保持简短。

### 4.3 application 层

`application/` 负责业务用例编排：

- 协调多个领域服务
- 调用模型客户端
- 调用工具
- 组织一次完整请求流程
- 控制事务边界
- 处理跨模块流程

application 层可以编排，但不应包含底层实现细节。

禁止：

- 直接写 SQL
- 直接依赖具体数据库驱动
- 直接读取环境变量
- 直接处理 HTTP Request 或 Response
- 将所有流程都塞进一个 service 文件

每个 application service 应对应明确用例，例如：

- 普通聊天
- 查询生产进度
- 分析工单风险
- 执行工具调用

### 4.4 domain 层

`domain/` 负责确定性业务规则：

- 业务实体
- 值对象
- 状态判断
- 风险规则
- 领域服务
- 业务异常

要求：

- 不依赖 FastAPI
- 不依赖数据库框架
- 不依赖 DeepSeek SDK
- 不读取环境变量
- 不感知 HTTP
- 规则必须可单元测试

生产进度、延期判断、状态流转等规则必须放在 domain 层，不允许交给大模型自由判断。

### 4.5 infrastructure 层

`infrastructure/` 负责外部系统实现：

- 数据库访问
- MES 接口调用
- DeepSeek API 调用
- 文件存储
- 日志实现
- 第三方服务客户端

建议按外部能力拆分：

- `infrastructure/llm/`
- `infrastructure/database/`
- `infrastructure/mes/`
- `infrastructure/repository/`

禁止：

- 将业务规则写入基础设施层
- 在数据库 Repository 中编排业务流程
- 让模型客户端直接调用数据库
- 让数据库代码依赖 FastAPI 路由

### 4.6 schemas 层

`schemas/` 负责：

- 请求 DTO
- 响应 DTO
- 工具参数模型
- 外部接口数据模型

要求：

- 使用 Pydantic 明确定义字段
- 禁止在接口之间传递无约束 `dict`
- 禁止使用语义不明确的通用字段
- 请求、响应、领域对象不要混用同一个模型

### 4.7 core 层

`core/` 负责基础能力：

- 配置读取
- 日志配置
- 通用异常
- 安全配置
- 应用常量

要求：

- 配置统一读取
- 环境变量不得散落在业务代码中
- 密钥只能从环境变量读取
- `.env.example` 只提供字段名和示例，不保存真实密钥

## 5. Python 文件拆分规则

AI 编写 Python 时，必须优先检查是否出现“功能揉在一个文件里”的问题。

出现以下任一情况，应考虑拆分：

- 单个文件同时包含路由、业务、数据库、模型调用
- 单个类承担多个无关职责
- 单个函数同时完成请求解析、查询、计算和响应拼装
- 同一文件中出现多个不同业务模块
- 文件长度持续增长且职责已经不单一
- 测试时无法独立替换外部依赖

强制规则：

- 一个文件只表达一个主要职责
- 一个 service 只负责一类业务用例
- 一个 repository 只负责一种数据来源或一类聚合
- 一个 client 只封装一个外部系统
- 一个规则类只处理一个明确规则集合
- 不创建 `utils.py` 收纳所有杂项
- 不创建 `common.py` 作为无边界垃圾桶
- 通用函数必须按具体语义命名和归类

同时禁止过度拆分：

- 不为一两行代码创建独立文件
- 不为尚未存在的业务创建目录
- 不创建只有接口没有实现价值的空抽象
- 不为了模仿大型项目而制造层级

判断标准：

- 职责是否单一
- 是否方便测试
- 是否降低耦合
- 是否真实被复用

## 6. Python 依赖方向

依赖方向必须保持：

`api -> application -> domain`

`application -> infrastructure abstraction`

`infrastructure -> external systems`

核心约束：

- domain 不依赖 infrastructure
- domain 不依赖 api
- application 不依赖 FastAPI Request/Response
- api 不直接依赖数据库实现
- 模型客户端不能直接访问 Repository
- Repository 不能调用模型
- 前端不能承载后端业务规则

如需解耦外部实现，可以在 application 或 domain 邻近位置定义明确接口，由 infrastructure 实现。

不要为所有类都创建接口，只在确实存在替换、测试或边界隔离需求时使用。

---

## 7. FastAPI 接口约束

接口统一使用 `/api` 前缀。

要求：

- 路由按模块拆分
- 请求和响应必须有明确 Schema
- 返回结构保持稳定
- 错误返回必须明确
- 参数必须校验
- 禁止直接返回数据库 ORM 对象
- 禁止把内部异常堆栈返回前端
- 禁止在接口中返回模型原始响应全文

接口命名应表达业务语义，例如：

- `/api/health`
- `/api/chat`
- `/api/production-progress/query`

避免：

- `/api/do`
- `/api/execute`
- `/api/test2`
- `/api/common`

---

## 8. 大模型调用约束

后续接入 DeepSeek 时必须遵守：

- 模型调用统一通过独立 LLM Client
- 业务代码不得直接依赖 DeepSeek SDK
- 模型名称、地址、Token、超时统一配置
- 模型响应必须校验
- 工具调用参数必须使用 Schema 校验
- 不允许模型生成 SQL 后直接执行
- 不允许模型绕过工具白名单
- 不允许模型直接修改 MES 数据
- 不允许模型自行决定业务状态
- 模型失败必须有明确错误和日志
- 必须记录耗时和 Token 使用量

大模型职责仅限：

- 理解用户意图
- 提取参数
- 选择白名单工具
- 汇总工具结果
- 生成自然语言说明

确定性规则必须由代码实现。

## 9. MES 数据访问约束

MES 数据接入必须优先只读。

优先级：

1. 调用 MES 已有只读接口
2. 使用只读数据库账号访问指定视图或表
3. 最后才考虑数据同步

要求：

- 禁止使用生产库高权限账号
- 禁止模型自由拼 SQL
- SQL 必须固定、参数化
- Repository 只负责数据读取
- 业务计算放在 domain 或 application
- 同一个业务指标只能有一个明确计算口径
- 页面与 Agent 使用同一数据口径
- 查询必须有范围限制
- 大批量查询必须分页
- 所有工具调用必须记录审计日志

---

## 10. 异常处理约束

禁止：

- 空 `except`
- `except Exception: pass`
- 捕获异常后返回成功
- 用 `None` 隐藏错误
- 将所有异常统一转换成模糊提示
- 在多层重复打印同一异常

要求：

- 业务异常和系统异常分开
- 在边界层转换错误
- 日志记录完整上下文
- 对外返回稳定错误码和可理解信息
- 不暴露密钥、SQL、内部路径和堆栈

---

## 11. 日志约束

日志必须记录：

- 请求标识
- 关键业务参数
- 外部调用耗时
- 模型调用耗时
- 工具名称
- 工具执行结果状态
- 异常类型

禁止：

- 打印 DeepSeek Token
- 打印数据库密码
- 打印完整用户隐私数据
- 打印超大模型原始响应
- 使用无意义日志，例如“进入方法”“执行成功”但没有上下文

日志应服务于定位问题，而不是增加噪声。

## 19. LangGraph / LangChain Agent 约束

LangGraph 只负责 Agent 流程编排和条件分支。

LangChain 只负责模型调用、Tool 定义和结构化输出能力。

要求：

- Agent 节点保持单一职责
- Catalog、Matcher、Executor、Graph、API 分离
- Graph State 不保存数据库 Session、FastAPI Request/Response、ORM 对象或 LLM Client
- LangGraph 不接管身份、反馈、Issue、聊天持久化或数据库事务
- Text-to-SQL 生成结果未经 Validator 不得执行

禁止：

- 将 Graph、Prompt、Tool、SQL 全部揉进一个文件
- 在 Agent 节点中直接读取环境变量
- 让模型返回任意 Tool 名称后直接执行
- 让 Tool 生成或执行自由 SQL
- 启用未明确要求的 Checkpointer、缓存、多 Agent 或自动重试循环

## 20. Documentation Language Policy

### 20.1 总体原则

MES Agent 文档同时服务于软件工程维护、MES 业务理解和 Agent 知识沉淀。语言必须按文档用途选择，不得简单统一为全中文或全英文。

默认策略：

- 技术文档英文优先。
- MES 业务文档中文优先。
- 代码、标识符、Schema 和协议统一使用英文。
- 同时包含工程结构和 MES 知识时，机器字段与技术术语使用英文，业务解释使用中文。
- 准确性和可维护性优先，不得为了节省 Token 强制使用英文。

所有新增文档必须先明确分类、主要服务对象和落盘目录。

### 20.2 Architecture 文档

- 目录：`docs/architecture/`
- 语言：英文优先
- 适用：系统架构、模块设计、Runtime 设计、Agent 设计、数据流设计、依赖关系
- 原因：保持架构术语与代码命名、工程边界一致

推荐标题：

- `Capability Runtime Architecture`
- `Execution Engine Design`
- `Agent Router Design`

`Capability`、`Runtime`、`Execution`、`Agent`、`Router`、`Reasoning`、`Trace`、`Audit` 等技术关键词保持英文，不翻译代码中的既有技术名称。

### 20.3 Business 文档

- 目录：`docs/business/`
- 语言：中文优先
- 适用：MES 业务事实、制造流程、工艺说明、业务术语、现场叫法、确定性业务规则
- 原因：业务知识主要来自中文制造现场，应优先保证领域人员准确理解和维护

推荐标题：

- `热处理追溯编号规则`
- `炉子、设备、炉次之间的业务关系`
- `热处理完成状态定义`

Business 文档中的代码值、字段名、API 路径、Enum 值和证据标识仍保持英文。

### 20.4 Capability 文档

- 目录：`docs/capabilities/`
- 语言：按字段用途混合使用
- 技术字段、机器字段、标识符：英文
- 业务描述、业务上下文、现场表达、示例：中文

示例：

```yaml
name: heat_current_stage
description: 查询热处理记录当前状态。
business_context:
  用户可能使用：
  - 做完了吗
  - 当前到哪一步
  - 是否结束
input_entities:
  - record_no
```

`name`、`description`、`business_context`、`input_entities`、`record_no` 等字段或标识符不得翻译为中文。

### 20.5 Prompt 文档

- 目录：`prompt/`
- 语言：根据主要输入语言和业务上下文决定
- 当前 MES Agent 默认：中文 Prompt

当前用户输入、Business Facts、MES 术语和现场表达主要为中文，因此 Prompt 默认使用中文；仅在明确的模型约束或接口契约要求下使用其他语言。

示例：

```text
你是一个 MES 智能助手。

请理解用户业务需求，选择合适的系统 Capability。
```

不得为了节省 Token 强制将 Prompt 改为英文。

### 20.6 API Contract 文档

- 语言：英文优先
- 适用：API Contract、Capability Contract、Input Schema、Output Schema、Execution Type、错误契约和协议边界
- 原因：接口属于工程边界，必须与代码和 Schema 保持一致

当 Contract 定义架构边界时，优先落盘到 `docs/architecture/`。表示真实 MES 输入的业务示例可以保留中文。

### 20.7 问题集与 Benchmark 文档

- 目录：`docs/business-scenarios/`
- 语言：中文
- 适用：问题集、Benchmark Case、验收场景、操作员表达、制造现场对话
- 原因：场景必须模拟真实制造现场，而不是翻译后的工程语言

示例：

- `这个热处理完成了吗？`
- `哪个炉子处理的？`
- `最近有没有异常？`

`id`、`expected_capability`、`expected_behavior` 等机器字段及枚举值仍保持英文。

### 20.8 代码命名规则

类名、函数名、方法名、变量名、常量名、包名、模块名、Schema 字段和协议字段统一使用英文。

正确：

- `HeatTreatmentAgent`
- `CapabilityRuntime`
- `BusinessFactLoader`

禁止：

- 中文类名、变量名或包名
- `ReChuLiAgent`
- `NengLiRuntime`

不得翻译代码中的既有技术名称，例如不得将 `Capability` 翻译后用于类名或协议字段。

### 20.9 Agent 知识文件

Business Facts：事实描述、业务含义、规则、边界和示例使用中文；代码符号、数据库名、字段名、Enum 值和证据标识使用英文。

Capability Catalog：字段与标识符使用英文；业务含义、`business_context` 内容、现场表达和示例使用中文。

示例：

```yaml
business_context: 热处理追溯编号用于定位热处理过程记录。
examples:
  - 这个批次做到哪一步了
  - 是否已经完成
```

### 20.10 新增文档判断清单

新增文档前必须判断：

1. 主要服务对象是谁？技术人员英文优先，业务人员中文优先。
2. 是否会被 LLM 读取？如会，业务内容优先保持最准确的现场表达。
3. 是否属于代码或协议契约？如是，英文优先。
4. 是否同时包含机器字段和业务解释？如是，字段英文、解释中文。
5. 是否已有同主题文档？如有，优先追加并保留历史时间线，不重复创建。

### 20.11 文档落盘规则

代码事实审查、业务事实审查、架构审查、技术调研、实验报告、Root Cause Analysis、Catalog Review、Gold Review 产生可复用结论时，必须落盘。

执行顺序：

1. 搜索 `docs/`、`doc/`、`knowledge/` 是否存在同主题文档。
2. 如存在，追加带日期的章节，保留历史结论与时间线。
3. 如不存在，按照本节分类规则在对应目录创建文档。
4. 如 `docs/` 不存在，创建 `docs/`。
5. 每次任务完成后，在 `log/codex-task-log.md` 追加任务目标、修改文件、关键决策、验证命令、验证结果、未完成项和风险。

禁止：

- 可复用结论仅保留在终端或聊天记录。
- 重复生成同主题文档。
- 创建 `final_v2_final_fix.md` 一类文件名链。
- 未说明例外便将文档放入未分类目录。
- 本可追加带日期章节时重写历史结论。

文档落盘后的输出回执必须包含：

- 文档路径。
- 新增章节。
- 追加内容摘要。

### 20.12 禁止的语言策略

禁止所有文档统一英文，因为 MES 业务知识整体英文化会降低现场准确性和维护效率。

禁止所有文档统一中文，因为工程契约和代码概念翻译后会与实现脱节。

禁止翻译既有代码名和技术标识，特别是不得将代码级 `Capability` 改为中文类名。

禁止在同一语义层随意混用语言。混合语言文档必须保持明确边界：机器与工程字段使用英文，MES 业务含义使用中文。

### 20.13 既有文档兼容规则

本规范约束新增文档和未来的实质性更新，不要求自动重写既有文档。既有文档发生实质性更新时，新增章节遵守本规范；完整语言迁移必须作为单独、明确范围的文档治理任务执行。
