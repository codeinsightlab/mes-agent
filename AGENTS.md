# AGENTS.md

## 1. 项目定位

本项目是一个独立的 MES Agent 研究项目。

当前技术栈：

- 前端：Vue 3 + Vite + JavaScript
- 后端：Python + FastAPI
- 模型：后续接入 DeepSeek
- 数据来源：后续通过只读接口或只读数据库访问 MES 数据
- 部署：本地开发优先，后续可使用 Docker Compose

核心原则：

- 先完成最小闭环，再扩展能力
- 数据正确性优先于智能化
- 业务规则必须确定、可验证
- 大模型只负责理解、编排和表达，不负责替代业务规则
- 禁止为未来需求提前设计复杂框架

---

## 2. 全局执行规则

任何任务开始前，必须先完成以下判断：

1. 当前需求是否明确
2. 修改范围是否清晰
3. 是否会影响现有功能
4. 是否需要新增依赖
5. 是否存在更简单的实现方式

执行时必须遵守：

- 只修改当前任务涉及的文件
- 不擅自扩大需求范围
- 不重构与当前任务无关的代码
- 不创建未被实际使用的抽象
- 不引入当前阶段不需要的中间件
- 不为了“以后可能使用”创建空模块
- 不隐藏错误，不吞异常
- 不伪造验证结果
- 不执行 Git commit
- 每次任务完成后更新 `log/codex-task-log.md`

## 4. Python 后端分层约束

Python 后端必须保持清晰分层。

推荐基础结构：

- `backend/app/main.py`
- `backend/app/api/`
- `backend/app/application/`
- `backend/app/domain/`
- `backend/app/infrastructure/`
- `backend/app/schemas/`
- `backend/app/core/`
- `backend/tests/`

各层职责如下。

### 4.1 main.py

`main.py` 只负责：

- 创建 FastAPI 应用
- 注册路由
- 注册中间件
- 注册生命周期事件
- 加载基础配置

禁止在 `main.py` 中：

- 编写业务逻辑
- 编写 SQL
- 调用大模型
- 实现 Agent 流程
- 处理复杂数据转换
- 定义大量接口

`main.py` 应保持短小，只作为应用入口。

### 4.2 api 层

`api/` 只负责 HTTP 协议相关内容：

- 接收请求
- 参数校验
- 调用 application 层
- 返回响应
- 转换 HTTP 状态码

禁止：

- 在路由函数中写业务判断
- 在路由函数中直接访问数据库
- 在路由函数中直接调用 DeepSeek
- 在路由函数中写复杂循环
- 在路由函数中拼装大段提示词
- 在路由函数中处理事务

路由函数应尽量保持简短。

### 4.3 application 层

`application/` 负责业务用例编排：

- 协调多个领域服务
- 调用模型客户端
- 调用工具
- 组织一次完整请求流程
- 控制事务边界
- 处理跨模块流程

application 层可以编排，但不应包含底层实现细节。

禁止：

- 直接写 SQL
- 直接依赖具体数据库驱动
- 直接读取环境变量
- 直接处理 HTTP Request 或 Response
- 将所有流程都塞进一个 service 文件

每个 application service 应对应明确用例，例如：

- 普通聊天
- 查询生产进度
- 分析工单风险
- 执行工具调用

### 4.4 domain 层

`domain/` 负责确定性业务规则：

- 业务实体
- 值对象
- 状态判断
- 风险规则
- 领域服务
- 业务异常

要求：

- 不依赖 FastAPI
- 不依赖数据库框架
- 不依赖 DeepSeek SDK
- 不读取环境变量
- 不感知 HTTP
- 规则必须可单元测试

生产进度、延期判断、状态流转等规则必须放在 domain 层，不允许交给大模型自由判断。

### 4.5 infrastructure 层

`infrastructure/` 负责外部系统实现：

- 数据库访问
- MES 接口调用
- DeepSeek API 调用
- 文件存储
- 日志实现
- 第三方服务客户端

建议按外部能力拆分：

- `infrastructure/llm/`
- `infrastructure/database/`
- `infrastructure/mes/`
- `infrastructure/repository/`

禁止：

- 将业务规则写入基础设施层
- 在数据库 Repository 中编排业务流程
- 让模型客户端直接调用数据库
- 让数据库代码依赖 FastAPI 路由

### 4.6 schemas 层

`schemas/` 负责：

- 请求 DTO
- 响应 DTO
- 工具参数模型
- 外部接口数据模型

要求：

- 使用 Pydantic 明确定义字段
- 禁止在接口之间传递无约束 `dict`
- 禁止使用语义不明确的通用字段
- 请求、响应、领域对象不要混用同一个模型

### 4.7 core 层

`core/` 负责基础能力：

- 配置读取
- 日志配置
- 通用异常
- 安全配置
- 应用常量

要求：

- 配置统一读取
- 环境变量不得散落在业务代码中
- 密钥只能从环境变量读取
- `.env.example` 只提供字段名和示例，不保存真实密钥

## 5. Python 文件拆分规则

AI 编写 Python 时，必须优先检查是否出现“功能揉在一个文件里”的问题。

出现以下任一情况，应考虑拆分：

- 单个文件同时包含路由、业务、数据库、模型调用
- 单个类承担多个无关职责
- 单个函数同时完成请求解析、查询、计算和响应拼装
- 同一文件中出现多个不同业务模块
- 文件长度持续增长且职责已经不单一
- 测试时无法独立替换外部依赖

强制规则：

- 一个文件只表达一个主要职责
- 一个 service 只负责一类业务用例
- 一个 repository 只负责一种数据来源或一类聚合
- 一个 client 只封装一个外部系统
- 一个规则类只处理一个明确规则集合
- 不创建 `utils.py` 收纳所有杂项
- 不创建 `common.py` 作为无边界垃圾桶
- 通用函数必须按具体语义命名和归类

同时禁止过度拆分：

- 不为一两行代码创建独立文件
- 不为尚未存在的业务创建目录
- 不创建只有接口没有实现价值的空抽象
- 不为了模仿大型项目而制造层级

判断标准：

- 职责是否单一
- 是否方便测试
- 是否降低耦合
- 是否真实被复用

## 6. Python 依赖方向

依赖方向必须保持：

`api -> application -> domain`

`application -> infrastructure abstraction`

`infrastructure -> external systems`

核心约束：

- domain 不依赖 infrastructure
- domain 不依赖 api
- application 不依赖 FastAPI Request/Response
- api 不直接依赖数据库实现
- 模型客户端不能直接访问 Repository
- Repository 不能调用模型
- 前端不能承载后端业务规则

如需解耦外部实现，可以在 application 或 domain 邻近位置定义明确接口，由 infrastructure 实现。

不要为所有类都创建接口，只在确实存在替换、测试或边界隔离需求时使用。

---

## 7. FastAPI 接口约束

接口统一使用 `/api` 前缀。

要求：

- 路由按模块拆分
- 请求和响应必须有明确 Schema
- 返回结构保持稳定
- 错误返回必须明确
- 参数必须校验
- 禁止直接返回数据库 ORM 对象
- 禁止把内部异常堆栈返回前端
- 禁止在接口中返回模型原始响应全文

接口命名应表达业务语义，例如：

- `/api/health`
- `/api/chat`
- `/api/production-progress/query`

避免：

- `/api/do`
- `/api/execute`
- `/api/test2`
- `/api/common`

---

## 8. 大模型调用约束

后续接入 DeepSeek 时必须遵守：

- 模型调用统一通过独立 LLM Client
- 业务代码不得直接依赖 DeepSeek SDK
- 模型名称、地址、Token、超时统一配置
- 模型响应必须校验
- 工具调用参数必须使用 Schema 校验
- 不允许模型生成 SQL 后直接执行
- 不允许模型绕过工具白名单
- 不允许模型直接修改 MES 数据
- 不允许模型自行决定业务状态
- 模型失败必须有明确错误和日志
- 必须记录耗时和 Token 使用量

大模型职责仅限：

- 理解用户意图
- 提取参数
- 选择白名单工具
- 汇总工具结果
- 生成自然语言说明

确定性规则必须由代码实现。

---

## 9. MES 数据访问约束

MES 数据接入必须优先只读。

优先级：

1. 调用 MES 已有只读接口
2. 使用只读数据库账号访问指定视图或表
3. 最后才考虑数据同步

要求：

- 禁止使用生产库高权限账号
- 禁止模型自由拼 SQL
- SQL 必须固定、参数化
- Repository 只负责数据读取
- 业务计算放在 domain 或 application
- 同一个业务指标只能有一个明确计算口径
- 页面与 Agent 使用同一数据口径
- 查询必须有范围限制
- 大批量查询必须分页
- 所有工具调用必须记录审计日志

---

## 10. 异常处理约束

禁止：

- 空 `except`
- `except Exception: pass`
- 捕获异常后返回成功
- 用 `None` 隐藏错误
- 将所有异常统一转换成模糊提示
- 在多层重复打印同一异常

要求：

- 业务异常和系统异常分开
- 在边界层转换错误
- 日志记录完整上下文
- 对外返回稳定错误码和可理解信息
- 不暴露密钥、SQL、内部路径和堆栈

---

## 11. 日志约束

日志必须记录：

- 请求标识
- 关键业务参数
- 外部调用耗时
- 模型调用耗时
- 工具名称
- 工具执行结果状态
- 异常类型

禁止：

- 打印 DeepSeek Token
- 打印数据库密码
- 打印完整用户隐私数据
- 打印超大模型原始响应
- 使用无意义日志，例如“进入方法”“执行成功”但没有上下文

日志应服务于定位问题，而不是增加噪声。

优先测试：

- 生产进度计算
- 延期风险规则
- 参数校验
- 工具参数解析
- 外部服务失败处理
- 边界值

- 异常状态

## 19. LangGraph / LangChain Agent 约束

LangGraph 只负责 Agent 流程编排和条件分支。

LangChain 只负责模型调用、Tool 定义和结构化输出能力。

要求：

- Agent 节点保持单一职责
- Catalog、Matcher、Executor、Graph、API 分离
- Graph State 不保存数据库 Session、FastAPI Request/Response、ORM 对象或 LLM Client
- LangGraph 不接管身份、反馈、Issue、聊天持久化或数据库事务
- Text-to-SQL 生成结果未经 Validator 不得执行

禁止：

- 将 Graph、Prompt、Tool、SQL 全部揉进一个文件
- 在 Agent 节点中直接读取环境变量
- 让模型返回任意 Tool 名称后直接执行
- 让 Tool 生成或执行自由 SQL
- 启用未明确要求的 Checkpointer、缓存、多 Agent 或自动重试循环
